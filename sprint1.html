[ZN-1089] Migrate package build/releaser jobs from old 'zonza-jenkins' box

James Young | james.young@bjss.com

Updates

Migrated jobs run in the last year

Using single Jenkins job with package parameter, keeps Jenkins maintainable and avoids clutter

Abstraction already done by the Makefile, Docker build container, and packaging-tools

Map of package names->repos kept in the Jenkinsfile, maintained by version control

Simple to add new packages to build

Road blocks

Makefiles have a lot of copy/pasted code, some of which makes bad assumptions about build environment

Easier to fix issues by fudging the build environment than updating many Makefiles in many repos

Use of explicitly configured web proxy in cloud is biggest headache - encounter connectivity issues again and again

Future work/lessons learned

Code reuse in Makefiles ("include") rather than copy/paste would make it easier to maintain

Adding transparent web proxying to core would fix a lot of build problems caused by connectivity
